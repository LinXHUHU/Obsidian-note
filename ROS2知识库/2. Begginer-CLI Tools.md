# 环境配置

 **ROS 2 是一个基于环境变量的生态系统。**

- **Underlay** = 官方基础。
- **Overlay** = 你的定制开发。
- **Source** = 开启这个生态系统的钥匙。

**一定要养成习惯**：每开始一个新的开发会话，确保你已经 source 了正确的 setup 文件！

_注意：不要在同一个终端里混合 source 两个不同版本的 ROS，这会导致库冲突和奇怪的错误。_


### `ROS_LOCALHOST_ONLY` 极简指南

**作用**：  
将 ROS 2 通信限制在**本机**。开启后，你的节点**无法**与局域网内的其他电脑/机器人通信，也看不到它们的数据。

**为什么用**：

- **防干扰**：在教室、实验室等多人环境中，防止误收别人的话题（如 `/cmd_vel`）导致机器人乱跑。
- **更干净**：调试时只看本机数据，避免网络广播噪音。

这部分教程介绍了 ROS 2 中一个非常强大的功能：**重映射 (Remapping)**。

### 🎯 核心问题

当你启动第二个键盘控制节点 (`turtle_teleop_key`) 时，默认情况下它和第一个节点一样，都会向同一个话题 **`/turtle1/cmd_vel`** 发送速度指令。

- **结果**：两个键盘都在控制 **乌龟1**，无法独立控制 **乌龟2**。

### 🛠️ 解决方案：话题重映射

我们需要告诉第二个控制节点：“把你原本要发给 `turtle1/cmd_vel` 的数据，改发给 `turtle2/cmd_vel`”。

#### 操作步骤

在新的终端中运行以下命令：

bash

编辑

```
ros2 run turtlesim turtle_teleop_key --ros-args --remap turtle1/cmd_vel:=turtle2/cmd_vel
```

#### 命令解析

- `--ros-args`：告诉 `ros2 run` 后面的参数是传给 ROS 2 系统的，而不是传给节点程序本身的。
- `--remap <原话题>:=<新话题>`：
    - `turtle1/cmd_vel`：节点内部默认发布的旧话题名称。
    - `:=`：重映射操作符。
    - `turtle2/cmd_vel`：你希望它实际发布到的新话题名称。

### ✅ 最终效果

现在你拥有了两套独立的控制系统：

1. **终端 A** (原始命令)：控制 **乌龟1** (发送到 `/turtle1/cmd_vel`)。
2. **终端 B** (重映射命令)：控制 **乌龟2** (发送到 `/turtle2/cmd_vel`)。

你可以分别激活这两个终端窗口，用方向键独立指挥两只乌龟移动，互不干扰。





这段文字介绍了 ROS 2 架构中最基础的两个概念：**ROS 图 (ROS Graph)** 和 **节点 (Nodes)**。

### 1. ROS 图 (The ROS 2 Graph)

- **定义**：它是所有正在运行的 ROS 2 元素及其相互连接关系的**整体网络视图**。
- **包含内容**：
    - 所有正在运行的可执行程序（节点）。
    - 它们之间的数据连接（话题、服务等）。
- **形象理解**：如果你把系统中所有的软件模块画成圆圈，把它们之间的通信画成连线，这张完整的“地图”就是 **ROS 图**。它展示了数据如何在系统中流动。

### 2. 节点 (Nodes in ROS 2)

- **定义**：构成 ROS 图的**基本构建单元**（即图中的每个圆圈）。
- **核心原则**：**单一职责 (Single, Modular Purpose)**。
    - 每个节点只负责一个具体的小任务。
    - _例子_：一个节点专门控制电机，另一个节点专门发布激光雷达数据，再一个节点专门处理图像。
- **通信方式**：节点之间不直接调用函数，而是通过以下机制交换数据：
    - **话题 (Topics)**：单向数据流（如传感器数据）。
    - **服务 (Services)**：请求/响应模式（如开关灯）。
    - **动作 (Actions)**：带反馈的长时任务（如导航到某点）。
    - **参数 (Parameters)**：配置信息（如设置速度上限）。

### 💡 总结

- **节点**是干活的“工人”，每人只专攻一项技能。
- **ROS 图**是整个“工厂”的布局图，展示了所有工人以及他们之间是如何传递消息协作的。

- **节点 (Node)** 是 ROS 2 系统的**基本构建单元**。
- **原则**：每个节点只负责**单一、模块化**的任务（例如：只负责显示乌龟，或只负责读取键盘输入）。

为了管理和调试复杂的机器人系统，你掌握了两个至关重要的命令行工具：

表格

|命令|作用|使用场景|
|---|---|---|

- `ros2 node list` 让你一眼看清**“谁在场”**。
- `ros2 node info` 让你深入理解**“它在做什么”**以及**“它如何与其他部分通信”**。
- 掌握这些工具是理解数据流向、排查故障和构建复杂系统的基础。




### 🚀 核心命令：`ros2 topic pub`

#### 1. 基本语法

bash

编辑

```
ros2 topic pub <话题名称> <消息类型> '<参数数据>'
```

- **`<话题名称>`**：你要发送数据的目标（例如 `/turtle1/cmd_vel`）。
- **`<消息类型>`**：数据的结构定义（例如 `geometry_msgs/msg/Twist`）。
- **`'<参数数据>'`**：具体的数值，必须符合消息类型的结构，且通常使用 **YAML 格式**。

#### 2. 实战案例：让乌龟动起来

为了让乌龟持续移动，你需要发送一个包含线速度（linear）和角速度（angular）的 `Twist` 消息。

**完整命令：**

bash

编辑

```
ros2 topic pub /turtle1/cmd_vel geometry_msgs/msg/Twist "{linear: {x: 2.0, y: 0.0, z: 0.0}, angular: {x: 0.0, y: 0.0, z: 1.8}}"
```

**命令解析：**

- **目标**：`/turtle1/cmd_vel`（控制乌龟1的速度）。
- **类型**：`geometry_msgs/msg/Twist`（表示三维空间中的速度和旋转）。
- **数据 (YAML)**：
    - `linear: {x: 2.0, ...}`：以 2.0 m/s 的速度向前移动（X轴）。
    - `angular: {z: 1.8}`：以 1.8 rad/s 的速度逆时针旋转（绕Z轴）。
    - _结果_：乌龟会一边向前走，一边转圈。

#### 3. 关键特性：自动循环发布 (1 Hz)

- **默认行为**：如果你不加任何额外参数，`ros2 topic pub` 会以 **1 Hz**（每秒1次）的频率持续发送该消息。
- **为什么需要持续发送？**
    - 真实的机器人（以及仿真中的乌龟）通常需要**持续的控制信号**来保持运动。
    - 如果只发送一次就停止，许多机器人的安全机制会让它立即停下，或者它只会动一下就停了。
    - 这个默认的“稳态流”（steady stream）确保了乌龟能持续执行你的指令。




`Header` 是 ROS 生态系统的**通用元数据容器**。

- **来源**：`std_msgs` 功能包。
- **作用**：统一携带 **时间 (`stamp`)** 和 **坐标系 (`frame_id`)**。
- **价值**：让所有带有时空属性的消息（带 `Stamped` 后缀的通常都有）能够被系统统一处理，是实现多传感器融合和坐标变换（TF2）的基石。



这段文字清晰地介绍了 ROS 2 中另一种核心的通信机制：**服务 (Services)**。

它与之前讨论的 **话题 (Topics)** 形成了鲜明的对比。为了帮助你更好地理解，我们可以从以下几个维度来解析：

### ⚔️ 核心对比：话题 (Topics) vs. 服务 (Services)

表格

|特性|话题 (Topics)|服务 (Services)|
|---|---|---|
|**通信模型**|**发布 - 订阅 (Pub/Sub)**  <br>(单向广播)|**调用 - 响应 (Call/Response)**  <br>(双向交互)|
|**数据流**|**持续流 (Stream)**  <br>像看电视直播，数据源源不断。|**一次性交互 (Request/Reply)**  <br>像打电话或问路，问一次答一次。|
|**触发方式**|发布者按自己的频率发送，订阅者被动接收。|只有当客户端 (Client) **主动调用**时，服务端 (Server) 才处理并返回数据。|
|**适用场景**|传感器数据 (激光雷达、摄像头)、状态更新 (里程计、电池电量)。|逻辑控制 (开关灯)、计算任务 (路径规划)、查询状态 (获取当前模式)、重置操作。|
|**实时性**|适合高频、低延迟的数据流。|适合低频、需要确认结果的任务。|
|**阻塞性**|非阻塞，订阅者不会卡住发布者。|同步调用会阻塞客户端直到收到响应（异步调用除外）|

### 💡 其他常见的 "触发型" 服务

除了 `/clear`，ROS 中还有很多类似的服务，它们通常也使用 `Empty` 或者类似的简单类型（如 `Trigger`）：

- `/reset`：重置仿真环境。
- `/start_recording`：开始录制数据。
- `/emergency_stop`：紧急停止机器人（只需触发，无需参数）。

### 结论

这段文字教会了你两件事：

1. 如何使用 `ros2 service type` 来检查服务的接口定义。
2. 理解了 **`Empty` 类型** 的存在意义：它是一种**纯粹的指令触发机制**，用于执行那些“不需要输入参数，也不需要返回数据，只要执行即可”的任务。


它是为了解决 **服务 (Services)** 在处理**长时间运行任务**时的局限性而设计的。

### 🚀 核心概念：为什么需要动作 (Actions)？

想象一下你让机器人去“从客厅走到厨房”：

- **如果用服务 (Service)**：
    
    - 你发送请求：“去厨房”。
    - 然后你就**死等了**（阻塞）。机器人走了一半，你不知道它走到哪了；如果路被堵住了，你也没法让它停下来，只能等它自己走完或者报错超时。
    - _缺点：无中间状态，不可取消，长时间阻塞。_
- **如果用动作 (Action)**：
    
    - 你发送目标：“去厨房”。
    - 机器人立刻回复：“收到，开始出发”。
    - **过程中**：机器人每隔一秒告诉你：“我走了10%... 30%... 50%... 遇到障碍物暂停了一下... 现在70%”。 (**反馈 Feedback**)
    - **中途**：你突然改变主意，可以发送指令：“别去了，回来！” (**可取消 Cancelable**)
    - **结束时**：机器人告诉你：“任务完成，我到了”或者“任务失败，电池没电了”。 (**结果 Result**)

### ⚙️ 动作的三大组成部分

动作通信由三个关键部分构成，这也是它与话题和服务的最大区别：

1. **目标 (Goal)**：
    
    - **方向**：客户端 -> 服务端。
    - **作用**：定义任务的目标（例如：导航坐标、机械臂抓取位置）。
    - _类似服务的请求 (Request)。_
2. **反馈 (Feedback)**：
    
    - **方向**：服务端 -> 客户端。
    - **作用**：在任务执行过程中，**持续不断地**发送进度信息。
    - _这是服务做不到的（服务只有最后才回一次话）。_
    - _类似话题的订阅 (Subscribe)，但是是专门针对这个任务的。_
3. **结果 (Result)**：
    
    - **方向**：服务端 -> 客户端。
    - **作用**：任务结束后发送的最终状态（成功/失败，以及最终数据）。
    - _类似服务的响应 (Response)。_

### 🔄 动作 vs. 服务 vs. 话题

表格

|特性|话题 (Topics)|服务 (Services)|**动作 (Actions)**|
|---|---|---|---|
|**模型**|发布/订阅 (多对多)|调用/响应 (一对一)|**客户端/服务端 (一对一)**|
|**数据流**|持续不断的数据流|单次请求 -> 单次响应|**目标 -> (持续反馈) -> 结果**|
|**执行时间**|实时流，无明确结束|短时任务，快速返回|**长时运行任务 (秒级/分钟级)**|
|**可取消性**|❌ 无法取消已发出的消息|❌ 一旦调用，必须等到结束|✅ **可以随时取消目标**|
|**进度反馈**|❌ 只是数据流，无任务概念|❌ 只有最后的结果|✅ **提供实时的进度反馈**|
|**底层实现**|DDS 直接传输|基于话题 (请求/响应话题)|**基于话题 (5个内部话题)**|

### 🏗️ 技术架构：动作是如何构建的？

文中提到 _"Actions are built on topics and services"_。  
实际上，一个动作服务器在后台会自动创建 **5 个内部话题** 来协同工作：

1. `.../goal` (话题): 客户端发送目标。
2. `.../cancel_goal` (话题): 客户端发送取消请求。
3. `.../status` (话题): 服务端发送目标状态（如：接受、执行中、成功、取消）。
4. `.../feedback` (话题): 服务端发送实时反馈。
5. `.../result` (话题): 服务端发送最终结果。

这种设计使得动作既拥有服务的**请求 - 响应逻辑**，又拥有话题的**异步流式传输能力**。

### 💡 典型应用场景

只要任务满足以下任一条件，就应该使用 **动作 (Action)** 而不是服务：

1. **任务耗时较长**（超过几百毫秒），不能让节点一直阻塞等待。
    - _例子：导航 (Navigate to Pose)、机械臂运动规划 (MoveIt)。_
2. **需要监控进度**。
    - _例子：显示“下载进度 50%”、“清洗地板进度 30%”。_
3. **可能需要中途打断**。
    - _例子：机器人正在去充电，突然有人呼叫它，需要立即停止充电任务去救人。_

### 📝 总结

- **Topics** = 直播流 (只管发，不管收，持续不断)。
- **Services** = 打电话问路 (问一次，答一次，说完就挂，不能中途插嘴)。
- **Actions** = 外卖订单 (下单 -> 骑手接单 -> **实时推送位置/进度** -> 送达/取消)。

动作是 ROS 2 中处理复杂、长期、交互式任务的标准解决方案。




ROS 2 的 **Action（动作）** 机制是其通信架构中最复杂但也最强大的部分。它并不是凭空产生的新协议，而是**基于 Topic（话题）和 Service（服务）的概念，通过一组特定的内部 Topic 组合而成的**。

简单来说：**Action = 1个目标请求 + 1个取消请求 + 1个状态反馈 + 1个结果返回 + 状态机管理**。

以下是 Action 实现的深层机制解析：

### 1. 核心架构：5 个内部 Topic

当你创建一个名为 `/navigate_to_pose` 的 Action Server 时，ROS 2 底层实际上会自动创建 **5 个具体的 Topic** 来处理通信。客户端和服务端通过这些 Topic 进行交互。

假设 Action 名称为 `<action_name>`，类型为 `<ActionType>`，底层会生成以下 Topic：

表格

|序号|Topic 名称后缀|方向|消息类型|作用|
|---|---|---|---|---|
|1|`/_action/goal`|Client → Server|`<ActionType>_Goal`|**发送目标**：客户端提交任务请求。|
|2|`/_action/cancel_goal`|Client → Server|`CancelGoal`|**请求取消**：客户端发送取消特定任务的请求。|
|3|`/_action/feedback`|Server → Client|`<ActionType>_Feedback`|**实时反馈**：服务端持续发送进度数据。|
|4|`/_action/status`|Server → Client|`GoalStatusArray`|**状态监控**：服务端广播所有活跃目标的状态（如：接受、执行中、成功、取消）。|
|5|`/_action/result`|Server → Client|`<ActionType>_Result`|**最终结果**：任务结束后发送一次性结果数据。|

> **注意**：你在命令行使用 `ros2 topic list` 时通常看不到这些话题，因为它们的名字被 Action 库“隐藏”或命名空间化了，需要通过 `ros2 action list` 来查看逻辑上的 Action。

### 2. 通信流程与状态机 (State Machine)

Action 的核心不仅仅是数据传输，更是一个**有限状态机 (Finite State Machine)** 的管理过程。每个 Goal（目标）都有独立的生命周期。

#### 典型生命周期流程：

1. **发送目标 (Send Goal)**:
    
    - Client 向 `/_action/goal` 发送请求。
    - Server 收到后，进入 **回调函数 (Goal Callback)**。
    - Server 决定是 **接受 (ACCEPT)** 还是 **拒绝 (REJECT)**。
    - 如果接受，Server 将该 Goal ID 加入活跃列表，并通过 `/_action/status` 广播状态为 `EXECUTING`。
2. **执行与反馈 (Execute & Feedback)**:
    
    - Server 在后台线程或执行器中执行任务。
    - 在执行过程中，Server 周期性地向 `/_action/feedback` 发布进度消息。
    - Client 订阅该话题，实时更新进度条或日志。
3. **取消请求 (Cancel Request) [可选]**:
    
    - 如果 Client 想中途停止，向 `/_action/cancel_goal` 发送包含 Goal ID 的取消请求。
    - Server 收到后，进入 **取消回调 (Cancel Callback)**。
    - Server 决定是 **同意取消** 还是 **拒绝取消**（例如任务已接近完成，不可中断）。
    - 如果同意，Server 停止任务，将状态更新为 `CANCELING` 最终变为 `CANCELED`。
4. **结束与结果 (Finish & Result)**:
    
    - 当任务完成（成功、失败或被取消），Server 向 `/_action/result` 发送最终结果消息。
    - Server 通过 `/_action/status` 更新状态为 `SUCCEEDED`、`FAILED` 或 `CANCELED`。
    - Client 收到结果后，触发完成回调，通信结束。

### 3. 代码层面的实现机制 (rclpy / rclcpp)

在开发者层面，你不需要手动创建这 5 个 Topic。ROS 2 的客户端库（`rclpy` 或 `rclcpp`）封装了复杂的逻辑。

#### A. 服务端 (Action Server)

- **注册**: 你实例化一个 `ActionServer` 对象，传入节点、Action 类型、Action 名称以及三个核心回调函数：
    1. `execute_callback`: 处理主要逻辑（必须实现）。
    2. `goal_callback`: 决定是否接受新目标（可选，默认接受）。
    3. `cancel_callback`: 决定是否允许取消（可选，默认允许）。
- **执行器 (Executor)**: Action Server 依赖 ROS 2 的 Executor 来轮询内部的 Topic。当你调用 `spin()` 时，Executor 会检查是否有新的 Goal、Cancel 请求，并调度相应的回调函数。
- **反馈发布**: 在 `execute_callback` 中，你通过 `goal_handle.publish_feedback()` 方法发送反馈，底层会自动将其序列化并发布到 `/_action/feedback` Topic。

#### B. 客户端 (Action Client)

- **异步非阻塞**: 调用 `send_goal_async()` 是非阻塞的。它立即返回一个 `Future` 对象。
- **回调链**:
    1. **Goal Response Callback**: 当 Server 回复“接受”或“拒绝”时触发。
    2. **Feedback Callback**: 每次收到 `/_action/feedback` 消息时触发。
    3. **Result Callback**: 当收到 `/_action/result` 且任务终结时触发。
- **取消机制**: 客户端持有 Goal ID，可以随时调用 `cancel_goal_async()` 发送取消请求。

### 4. 关键特性解析

#### 为什么需要 UUID (Goal ID)?

每个 Action 目标都有一个唯一的 UUID。

- **多任务并发**：一个 Action Server 可以同时处理多个 Goal（例如机器人同时规划多条路径，虽然通常一次只执行一个，但可以排队）。
- **精准取消**：`cancel_goal` 消息中必须包含 UUID，确保只取消特定的任务，而不是取消所有正在运行的任务。

#### 可靠性 (QoS)

底层的 5 个 Topic 可以配置不同的 QoS 策略：

- `goal` 和 `result` 通常需要 **Reliable**（可靠传输），确保任务指令和结果不丢失。
- `feedback` 有时可以配置为 **Best Effort**（尽力而为），因为旧的进度数据丢失了没关系，只要最新的能收到即可（取决于具体应用）。

### 5. 总结：Action 的本质

表格

|层面|描述|
|---|---|
|**物理层**|基于 DDS/RTPS 的 **5 个 Topic** 数据流。|
|**逻辑层**|一个带有 **状态机 (State Machine)** 的长时任务管理器。|
|**功能层**|结合了 **Service 的请求 - 响应** 模式 和 **Topic 的流式反馈** 能力，并增加了 **可取消性**。|
|**适用场景**|导航、机械臂运动、长时间数据采集、任何需要“进度条”和“紧急停止”的任务。|

**一句话总结**：  
Action 机制是通过在底层自动管理一组特定的 Topic，并在应用层维护一个带 ID 的状态机，从而实现了对长时任务的**可控（可取消）、可视（有反馈）、可信（有结果）**的执行管理。
