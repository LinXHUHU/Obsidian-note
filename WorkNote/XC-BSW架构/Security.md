# 密钥注入

### **Key Injection 的核心概念**
**Key Injection** 是将加密密钥安全地写入设备存储介质的过程，通常在制造阶段完成，目的是为后续的安全功能（如安全启动、加密通信、数据保护等）建立信任根。其核心流程包括：
1. **密钥生成**：在安全环境中生成密钥（如根密钥、派生密钥）。
2. **传输加密**：通过加密通道（如 TLS、安全 CAN 通信）将密钥传输到目标设备。
3. **写入安全存储**：将密钥写入设备的安全存储介质（如 RPMB、eFuse、Keystore）。
4. **权限锁定**：关闭写入接口或锁定存储区域，防止后续修改。

### **存储介质对比与选择**
密钥的存储介质直接影响其安全性和灵活性，以下是三种常见存储介质的对比：

| **存储介质**                                | **技术特点**                                                             | **安全性** | **适用场景**                        |
| --------------------------------------- | -------------------------------------------------------------------- | ------- | ------------------------------- |
| **RPMB（Replay Protected Memory Block）** | - UFS 闪存的专用安全分区<br>- 支持防重放攻击的读写操作<br>- 需 MAC 验证每次写入                  | 高       | 动态密钥（如会话密钥、计数器）、需更新的证书          |
| **eFuse（电子熔丝）**                         | - 一次性可编程（OTP）非易失性存储<br>- 物理不可逆，写入后无法修改<br>- 容量小（通常几百位）               | 极高      | 根密钥（Root Key）、硬件唯一密钥（HUK）、设备锁状态 |
| **Keystore（密钥存储区）**                     | - NOR Flash 或其他存储介质的专用分区<br>- 需软件或硬件（如 TrustZone）保护访问权限<br>- 支持多密钥存储 | 中高      | 派生密钥、应用层密钥、非根密钥                 |

---

### **Key Injection 的实现流程**
以下是一个典型的密钥注入流程（基于 UDS 诊断服务）：

1. **进入制造模式**  
   - 设备通过物理触发（如 JTAG 模式）或软件指令（如 Bootloader 特殊命令）进入允许密钥写入的特殊模式。

2. **身份认证**  
   - 通过安全认证（如基于证书的认证或预共享密钥）验证注入设备的身份，防止未经授权的密钥注入。

3. **传输密钥**  
   - 使用 UDS 诊断服务（如 `0x2E` 写内存、`0x36` 下载 + `0x37` 传输退出）将密钥传输到设备。

4. **写入安全存储**  
   - 根据配置选择存储介质（RPMB、eFuse 或 Keystore）：
     - **RPMB**：计算 MAC 并验证后写入。
     - **eFuse**：烧录 OTP 区域，需严格校验数据完整性。
     - **Keystore**：加密后写入 NOR Flash，并配置访问权限。

5. **权限锁定**  
   - 锁定写入接口（如禁用 JTAG、关闭 Bootloader 调试模式）。
   - 对于 eFuse，烧录锁定标志位以防止后续修改。

---

### **Key Injection 的安全挑战与对策**
1. **密钥泄露风险**  
   - **对策**：在安全环境中生成和传输密钥，使用加密通道（如 TLS 或 CAN FD SecOC）保护传输过程。

2. **防重放攻击**  
   - **对策**：在 RPMB 中使用防重放计数器（Replay Counter），每次写入需递增计数器并验证 MAC。

3. **物理攻击**  
   - **对策**：eFuse 和 TrustZone 等硬件安全机制可抵御物理探测和侧信道攻击。

4. **制造流程漏洞**  
   - **对策**：严格控制制造环境，使用自动化工具（如 Bosch DFlash 工具链）减少人为干预。

---

### **Key Injection 的应用场景**
1. **安全启动（Secure Boot）**  
   - 注入根密钥（Root Key）用于验证启动镜像的签名，确保只有授权固件可运行。

2. **加密存储（Encrypted Storage）**  
   - 注入文件系统加密密钥（如 AES-256 密钥），保护用户数据或日志文件。

3. **安全通信（Secure Communication）**  
   - 注入 TLS 证书或 CAN FD SecOC 的新鲜值（Freshness Value），用于车辆内外的加密通信。

4. **硬件绑定（Device Binding）**  
   - 注入硬件唯一密钥（HUK），用于绑定设备身份与授权服务（如 OTA 许可证）。

---
### **总结与建议**
- **选择存储介质**：根据密钥类型和需求选择存储介质：
  - 根密钥 → **eFuse**（不可逆，高安全性）。
  - 动态密钥 → **RPMB**（可更新，支持防重放）。
  - 应用层密钥 → **Keystore**（灵活，需软件保护）。
- **流程设计**：确保密钥注入流程的自动化和审计能力，减少人为干预。
- **安全防护**：结合硬件安全模块（HSM）、TrustZone 和加密通信技术，构建纵深防御体系。

如果你正在开发车载安全系统，建议：
1. 在制造阶段集成 Key Injection 工具链（如 Vector CANoe、Bosch DFlash）。
2. 使用 UDS 诊断服务（如 0x2E 写内存）实现标准化的密钥注入接口。
3. 在测试模式下通过 RPMB 或 Keystore 模拟密钥注入，验证流程安全性。

| 安全目标 | 实现方式 |
|----------|-----------|
| **完整性保护** | 使用 SHA-256 计算升级包哈希 |
| **身份认证** | 使用非对称签名（RSA/ECDSA），验证发布者身份 |
| **抗篡改能力** | 签名验证由 HSM 完成，防侧信道攻击 |
| **密钥管理** | 公钥按生命周期切换（Bosch/OEM） |
| **可信执行环境** | 使用 TrustZone 或 HSM 保证安全边界 |

---

## ✅ 四、可能的扩展方向

| 扩展方向          | 说明                                                    |
| ------------- | ----------------------------------------------------- |
| **多级签名机制**    | 支持 Root CA -> Intermediate CA -> Firmware Signer 的信任链 |
| **差分升级支持**    | 在签名验证前先解压/合并差分包                                       |
| **远程密钥更新机制**  | 支持 OTA 更新 OEM 公钥（需严格授权）                               |
| **日志审计与回滚机制** | 验证失败或升级失败记录到 Secure Log，并支持回滚至上一版本                    |
