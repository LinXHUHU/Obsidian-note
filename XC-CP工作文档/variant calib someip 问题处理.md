
# 背景

由于从 VRTE 2304 版本切换至 R2408 版本，导致当前 SOME/IP 通信中断，需要重新进行配置。  
初步检查发现，`portmap` 配置缺失，`comclient` 文件未安装到配置文件夹中，因此进行了相关配置和安装操作。
![[XC-CP工作文档/asset/Pasted image 20250411104309.png]]
# 解决方案

## 步骤一

首先确认 `portmap` 配置是否正确并完成配置。  
同时检查发现，`comclient` 文件尚未被安装到配置文件夹中，完成安装后继续调试。

调试过程中发现，只有 **Slave ORIN** 成功发送出了 SOME/IP offer 报文，这显然不符合预期。  
![[XC-CP工作文档/asset/Pasted image 20250411110005.png]]
---

## 步骤二

通过查看配置文件，发现 SOME/IP Server 的版本号与 Client 的版本号不一致，并且 Client 所需的版本号也与 Server 的版本号存在冲突。  
将两边的版本号统一为 `V1.0` 后，重新测试更改结果。
![[XC-CP工作文档/asset/Pasted image 20250411110027.png]]
然而，测试结果显示更改并未生效。

---

## 步骤三

寻求 Neero 专家的帮助，进一步排查问题。

### 检查端口占用情况
执行以下命令检查端口占用情况：
```bash
netstat -an
```
同时检查 ORIN-X 设备中的防火墙配置，查看 `/etc/fsi_bosch.yaml` 文件是否存在允许通过的规则：
```bash
cat /etc/fsi_bosch.yaml
```

结果显示，端口占用情况和防火墙配置均正常。

### 检查代码逻辑
专家团队进一步检查代码，发现问题是由于 **Instance ID** 配置不正确引起的。  
将 Instance ID 修改为 `someip:18` 类型，并遵循以下规则：
![[XC-CP工作文档/asset/Pasted image 20250411104738.png]]

测试修改后的代码，但问题依旧未能解决。

---

## 步骤四

经过多次尝试后，最终将 SOME/IP 版本号从 `V1.0` 更改为 `V1.1`，成功解决了核心崩溃的问题。

这一解决方案令人感到意外，因为通常版本号的降级可能会导致兼容性问题，但此次升级反而解决了问题。

---

## 步骤五

更改版本号后，SOME/IP Offer 报文能够成功发送，但在远程 Slave 设备上进行服务查找时仍然失败。  

### 原因分析
经过排查，发现是由于 **ECU-CFG 环境变量** 未正确设置。  
通过手动导出相关环境变量（例如 `export`），问题得以解决。

然而，在 Master 设备中使用本地方式进行服务查找时，依然失败。  
目前暂未明确具体原因，需进一步深入排查。

---

### 总结
1. 初步配置问题导致 SOME/IP 通信中断。
2. 版本号不一致及 Instance ID 配置错误是关键问题点。
3. 将版本号从 `V1.0` 升级为 `V1.1` 意外地解决了核心崩溃问题。
4. 远程 Slave 设备的服务查找失败由环境变量问题引起，通过导出变量得以解决。
